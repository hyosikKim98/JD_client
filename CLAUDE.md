# CLAUDE.md

이 문서는 「뭐 했음 청년」 React Native 클라이언트를 설계·구현·검토할 때 따라야 하는 작업 규칙이다.

이 문서의 제품 기준은 [`Plan_JD.md`](./Plan_JD.md)뿐이다. 서버 구현 문서나 아직 확정되지 않은 API를 추측하여 제품 동작을 임의로 바꾸지 않는다.

## 1. 작업 우선순위

충돌이 생기면 다음 순서로 판단한다.

1. 현재 사용자가 명시한 요청
2. 이 `CLAUDE.md`
3. `Plan_JD.md`의 제품 목표와 MVP 범위
4. 확정된 화면 설계, 디자인 시스템, API 명세
5. 기존 클라이언트 코드의 일관된 패턴

상위 기준과 기존 코드가 충돌하면 기존 코드를 그대로 확장하지 말고 충돌 내용과 변경 이유를 기록한다.

## 2. 제품 경험 원칙

모든 화면과 상호작용은 다음 원칙을 지켜야 한다.

- 사용자는 가입 직후 그룹 없이 개인 달력에서 기록할 수 있어야 한다.
- 첫 기록 전에 그룹 생성이나 초대 참여를 강요하지 않는다.
- 기록의 최소 필수 입력은 스티커 하나다.
- 소감과 사진은 선택 사항이며 입력하지 않아도 저장할 수 있다.
- 기록하지 않은 날을 실패처럼 표현하지 않는다.
- 벌금, 순위, 경쟁, 의무 인증을 연상시키는 UI를 추가하지 않는다.
- 스트릭은 압박 수단이 아니라 쌓인 성취를 보여주는 보조 정보로 다룬다.
- 빈 달력에는 경고 대신 오늘 날짜와 “스티커 하나만” 행동을 부드럽게 안내한다.
- 그룹 화면은 누가 기록했는지 쉽게 보여주되 기록하지 않은 사람을 부각하지 않는다.
- 사진 업로드 실패 때문에 스티커 기록 전체가 사라져서는 안 된다.
- 본인 기록에는 수정·삭제를, 다른 사람의 기록에는 반응만 제공한다.
- “잘 쉬었음”도 정상적인 성취로 취급한다.
- 사용자가 오류를 복구할 수 있도록 구체적이고 비난하지 않는 메시지를 제공한다.

새 기능을 제안하거나 구현하기 전에 다음 질문을 확인한다.

> 이 기능이 “강제 없이 가벼운 기록과 그룹 응원으로 지속할 수 있는가?”라는 MVP 검증에 필요한가?

아니라면 별도 요청 전까지 구현하지 않는다.

## 3. MVP 범위

### 포함

- 자체 회원가입·로그인
- 확정된 공급자의 소셜 로그인
- 개인 월간 달력
- 추천·커스텀 스티커
- 하루 스티커 기록 생성·수정·삭제
- 선택적 한 줄 소감과 사진
- 현재 스트릭과 방어권 표시
- 그룹 생성, 초대, 참여
- 그룹 공유 달력
- 멤버 목록과 최장 기록
- 고정 이모지 5종 반응
- 다중 그룹, 통합 달력, 기본 달력, 그룹 전환
- MVP 막바지의 선택 가능한 가벼운 리마인더

### 제외

별도 요청이 없으면 다음 기능을 만들지 않는다.

- 목표 설정 계층
- 주간·월간 통계 대시보드
- 회고 자동 생성
- 공유 이미지
- 배지와 경쟁형 보상
- AI 활동 요약
- 댓글과 채팅
- 신고·모더레이션 UI
- 공개 범위 세분화
- 실시간 WebSocket
- 복잡한 오프라인 동기화

## 4. 기본 기술 선택

새 클라이언트 프로젝트이고 별도 요구가 없다면 다음을 기본으로 한다.

- React Native
- TypeScript strict mode
- Expo managed workflow
- Expo Router
- TanStack Query: 서버 상태
- Zustand 또는 Context: 꼭 필요한 소규모 클라이언트 전역 상태
- React Hook Form: 폼 상태
- Zod: 폼과 API 경계 검증
- Expo SecureStore: 인증 토큰
- React Native Testing Library + Jest
- Maestro 또는 프로젝트에 확정된 도구: 핵심 모바일 E2E

규칙:

- 패키지 버전은 프로젝트 생성 시점의 호환 가능한 안정 버전을 사용한다.
- Expo SDK와 React Native의 공식 호환 범위를 벗어나지 않는다.
- 이미 프로젝트에 확정된 동등한 라이브러리가 있으면 중복 도입하지 않는다.
- 단순한 유틸리티를 위해 큰 라이브러리를 추가하지 않는다.
- 네이티브 모듈 추가 전 iOS·Android, Expo 개발 빌드, 배포 파이프라인 영향을 확인한다.
- 라이브러리는 유지보수 상태, 라이선스, 번들 영향, 테스트 가능성을 확인한 뒤 추가한다.

`Plan_JD.md`의 순서를 유지하며 각 단계가 실제 기기에서 독립적으로 동작하게 만든다.

4. 그룹

## 6. 작업 절차

모든 변경은 아래 순서로 수행한다.

1. `Plan_JD.md`에서 관련 사용자 흐름, 예외, 권한을 확인한다.
2. 현재 화면, 라우트, API 타입, 공통 컴포넌트, 테스트를 먼저 조사한다.
3. 한 문장으로 작업 범위와 사용자 완료 조건을 정한다.
4. 다음 상태를 빠짐없이 정의한다.
   - 최초 로딩
   - 성공
   - 빈 데이터
   - 입력 오류
   - 서버 오류
   - 네트워크 끊김
   - 권한 없음
   - 재시도
5. API 계약과 화면 상태를 먼저 정리한 뒤 구현한다.
6. 비즈니스 흐름, 접근성, iOS·Android 차이를 함께 구현한다.
7. 단위·컴포넌트·핵심 흐름 테스트를 추가한다.
8. 타입 검사, 린트, 테스트, 실제 앱 실행을 확인한다.
9. 사용자 흐름이나 API 계약이 바뀌면 관련 문서를 같은 작업에서 갱신한다.

화면 하나를 크게 만든 뒤 나누지 않는다. 달력, 기록 폼, 그룹 참여처럼 독립 테스트 가능한 단위로 나누어 구현한다.

## 7. 프로젝트 구조

Expo Router를 사용한다면 라우트와 기능 구현을 분리한다.

```text
app/
├── _layout.tsx
├── (public)/
│   ├── sign-in.tsx
│   └── sign-up.tsx
├── (protected)/
│   ├── _layout.tsx
│   ├── index.tsx
│   ├── calendar/
│   ├── records/
│   ├── groups/
│   ├── stickers/
│   └── settings/
└── invite/

src/
├── features/
│   ├── auth/
│   ├── calendar/
│   ├── record/
│   ├── group/
│   ├── membership/
│   ├── sticker/
│   ├── reaction/
│   ├── streak/
│   └── reminder/
├── shared/
│   ├── api/
│   ├── components/
│   ├── hooks/
│   ├── storage/
│   ├── theme/
│   ├── types/
│   └── utils/
└── test/
```

규칙:

- `app/` 파일은 라우트 연결과 화면 조합을 담당한다.
- 실제 기능 로직과 재사용 UI는 `src/features`에 둔다.
- 둘 이상의 기능에서 안정적으로 재사용되는 코드만 `src/shared`에 둔다.
- 기능 내부에는 필요에 따라 `api`, `components`, `hooks`, `model`, `screens`, `utils`를 둔다.
- 기능 간 내부 파일을 깊은 상대 경로로 직접 가져오지 않는다.
- 순환 의존이 생기면 공통 폴더로 옮기기 전에 기능 경계를 다시 검토한다.
- 모든 폴더에 의미 없는 `index.ts`를 만들지 않는다. 공개 API가 필요한 경계에서만 사용한다.

## 8. 상태 관리

상태의 소유권을 명확히 나눈다.

### 서버 상태

다음은 TanStack Query에서 관리한다.

- 사용자 정보
- 달력과 기록
- 그룹과 멤버
- 스티커
- 스트릭과 방어권
- 반응
- 초대 미리보기

규칙:

- 서버 응답을 Zustand나 Context에 복제하지 않는다.
- Query key는 기능별 factory로 중앙 관리한다.
- mutation 성공 후 무조건 전체 캐시를 비우지 않는다.
- 영향을 받은 월, 그룹, 기록 상세, 스트릭 query만 갱신하거나 무효화한다.
- 서버가 데이터의 최종 진실이다. 낙관적 UI는 실패 시 정확히 롤백 가능한 경우에만 사용한다.
- 반응 토글은 낙관적 업데이트가 가능하지만 중복 탭을 제어하고 실패 시 원상복구한다.
- 기록 생성·수정과 그룹 참여는 성공 응답 전에 완료된 것처럼 표시하지 않는다.

### 화면 로컬 상태

다음은 가능한 한 해당 컴포넌트나 화면에서 관리한다.

- 바텀 시트 열림 여부
- 선택 중인 스티커
- 입력 중인 소감
- 선택했지만 업로드하지 않은 사진
- 펼친 달력 날짜
- 확인 다이얼로그 상태

### 클라이언트 전역 상태

전역 상태는 최소화한다.

- 인증 세션의 메모리 상태
- 현재 선택한 달력처럼 여러 화면에서 공유해야 하는 UI 상태
- 아직 서버에 저장하지 않은 짧은 사용자 설정

Context나 Zustand를 서버 캐시 대체 용도로 사용하지 않는다.

## 9. API 통신 규칙

- API 호출은 공통 클라이언트를 통해 수행한다.
- 화면 컴포넌트에서 직접 `fetch`를 호출하지 않는다.
- 인증 헤더, 기본 URL, timeout, 공통 오류 변환은 API 계층에서 처리한다.
- API DTO와 화면 ViewModel을 구분한다.
- OpenAPI 명세가 있으면 타입과 클라이언트 생성을 우선 검토한다.
- 생성 타입을 직접 수정하지 않고 adapter에서 앱 모델로 변환한다.
- OpenAPI가 없다면 DTO를 기능의 `api` 폴더 한곳에 정의하고 화면마다 중복 선언하지 않는다.
- 서버 응답은 신뢰하지 않고 앱 경계에서 필요한 최소 형태를 검증한다.
- 자연어 오류 메시지가 아니라 안정적인 서버 오류 코드를 기준으로 분기한다.
- 알 수 없는 오류 코드에는 안전한 공통 메시지와 재시도 수단을 제공한다.
- 서버의 내부 오류 메시지나 stack trace를 사용자에게 표시하지 않는다.

오류는 최소 다음 범주로 정규화한다.

```ts
type AppError =
  | { type: 'network'; retryable: true }
  | { type: 'unauthorized'; retryable: boolean }
  | { type: 'forbidden'; retryable: false }
  | { type: 'validation'; fields?: Record<string, string> }
  | { type: 'conflict'; code: string }
  | { type: 'notFound'; code?: string }
  | { type: 'server'; retryable: true }
  | { type: 'unknown'; retryable: false };
```

규칙:

- GET 요청만 제한적으로 재시도한다.
- mutation을 자동 재시도할 때는 API의 멱등성이 확인되어야 한다.
- 네트워크 요청 중 버튼을 무조건 비활성화하는 것으로 끝내지 말고 진행 상태를 표시한다.
- 빠른 연속 탭으로 같은 기록이나 그룹이 중복 생성되지 않게 한다.

## 10. 인증과 세션

- 액세스 토큰과 리프레시 토큰을 일반 AsyncStorage에 저장하지 않는다.
- 인증 비밀값은 Expo SecureStore 등 OS 보안 저장소를 사용한다.
- 메모리에는 필요한 동안만 액세스 토큰을 유지한다.
- 토큰 값을 로그, 분석 이벤트, 오류 리포트에 포함하지 않는다.
- 여러 API 요청이 동시에 `401`을 받아도 토큰 갱신 요청은 하나만 실행한다.
- 토큰 갱신 성공 후 대기 중인 요청을 한 번만 재시도한다.
- 갱신 실패 시 세션과 사용자별 query cache를 제거하고 로그인 화면으로 이동한다.
- 로그아웃 시 보안 저장소, 사용자별 캐시, 선택 달력 상태, 임시 사진을 정리한다.
- 앱 시작 시 토큰 존재만으로 인증 완료 화면을 먼저 보여주지 않는다. 세션 복구 상태를 별도로 표시한다.
- 인증 라우트와 보호 라우트의 접근 제어를 최상위 layout에서 일관되게 처리한다.
- 비밀번호 입력은 OS 자동완성과 보안 입력 속성을 올바르게 설정한다.
- 소셜 로그인 공급자는 제품 결정 전 임의로 추가하지 않는다.

## 11. 화면과 내비게이션

### 필수 화면

- 로그인
- 회원가입
- 개인 달력 홈
- 기록 작성 바텀 시트
- 기록 상세
- 그룹 공유 달력
- 그룹 생성
- 그룹 참여 미리보기
- 멤버 목록
- 스티커 관리
- 설정과 리마인더

### 내비게이션 규칙

- 로그인 후 첫 진입은 사용자의 기본 달력이다.
- 기본 달력이 없거나 불러오지 못하면 개인 달력을 우선 복구한다.
- 그룹 참여 링크는 로그인 전에도 수신할 수 있어야 한다.
- 로그인 전 받은 초대는 인증 후 미리보기 화면으로 복원한다.
- 그룹 참여는 규칙 미리보기와 명시적 참여 버튼을 거친다.
- 뒤로 가기로 작성 중인 스티커·소감·사진이 사라질 때는 실제 입력이 있는 경우에만 확인한다.
- 같은 상세 화면을 반복 push하지 않는다.
- 삭제 후에는 존재하지 않는 상세 화면을 history에 남기지 않는다.
- 라우트 파라미터는 ID와 날짜 등 최소 값만 전달하고 전체 객체를 전달하지 않는다.

## 12. 달력 구현

달력은 제품의 핵심 화면이므로 외형보다 날짜 정확성과 정보 밀도를 우선한다.

### 날짜 규칙

- 기록 날짜의 앱 내부 표현은 `YYYY-MM-DD` 문자열을 기본으로 한다.
- 날짜 전용 값을 `new Date('YYYY-MM-DD')`로 파싱해 UTC 변환하지 않는다.
- API의 날짜 전용 필드를 timestamp로 취급하지 않는다.
- “오늘”은 제품에서 합의한 비즈니스 타임존을 기준으로 계산한다.
- 별도 결정 전 MVP 기준 타임존은 `Asia/Seoul`로 가정하고 한곳의 date adapter에 격리한다.
- 월 이동, 월말, 연말, 윤년을 라이브러리 또는 검증된 date utility로 처리한다.
- locale은 한국어를 기본으로 하되 문자열을 컴포넌트에 흩뿌리지 않는다.

### 개인 달력

- 날짜별 내 스티커를 보여준다.
- 오늘 칸을 시각적으로 강조한다.
- 기록이 없는 오늘을 선택하면 기록 작성 흐름으로 바로 연결한다.
- 기록이 있으면 수정 모드 또는 상세로 이동한다.
- 현재 보고 있는 달력 기준 스트릭과 방어권만 상단에 표시한다.

### 그룹 공유 달력

- 날짜 칸에는 `(스티커)(멤버 이름)`을 세로로 나열한다.
- 제한된 높이를 넘으면 일부만 보여주고 펼치기 동작을 제공한다.
- 기록하지 않은 멤버 명단을 별도로 표시하지 않는다.
- 아무도 기록하지 않은 날은 일반 빈 칸으로 표현한다.
- 항목 탭 시 해당 기록 상세로 이동한다.

### 통합 달력

- 각 기록에 출처 그룹명 또는 “나의 기록”을 표시한다.
- 같은 날짜 여러 그룹 기록을 누락하거나 하나로 합치지 않는다.
- 그룹 필터와 기본 달력 변경은 명확히 구분한다.

### 성능

- 월 단위 데이터만 요청하고 화면에서 전체 기록 이력을 필터링하지 않는다.
- 날짜 셀에 불필요한 네트워크 hook을 각각 두지 않는다.
- 한 번의 월 query 결과를 화면용 map으로 변환한다.
- 달력 셀 key에 배열 index를 사용하지 않는다.
- 렌더링 최적화는 측정 근거가 있을 때 적용하되, 날짜 셀 props는 안정적으로 유지한다.

## 13. 기록 작성 흐름

- 기록 작성은 바텀 시트 또는 동등하게 빠른 모바일 UI로 제공한다.
- 스티커 선택만으로 저장 버튼을 활성화한다.
- 소감과 사진에는 필수 표시를 하지 않는다.
- 선택 가능한 스티커는 현재 기록 대상 그룹에 속한 것만 보여준다.
- 같은 날짜에 기록이 있으면 새 기록 화면이 아니라 수정 모드로 연다.
- 수정 모드에서는 기존 스티커, 소감, 사진을 미리 채운다.
- 기록 날짜와 대상 그룹을 사용자가 착각하지 않도록 상단에 표시한다.
- 저장 중 중복 탭을 방지하고 진행 상태를 보여준다.
- 저장 성공 후 해당 월 달력, 기록 상세, 스트릭 데이터를 갱신한다.
- 수정 성공 후 이전 데이터가 잠깐 다시 보이지 않도록 cache update 순서를 관리한다.
- 삭제는 본인 기록에서만 제공하고 확인 단계를 거친다.
- 삭제 문구는 스트릭 영향을 중립적으로 안내하되 사용자를 압박하지 않는다.

### 입력 규칙

- 스티커는 필수다.
- 소감 길이 제한은 서버 계약과 동일하게 적용한다.
- 클라이언트 검증은 편의를 위한 것이며 서버 검증을 대체하지 않는다.
- 키보드가 입력 필드와 저장 버튼을 가리지 않게 한다.
- 제출 후 검증 오류가 있으면 첫 오류 필드로 이동하거나 읽어준다.
- 사용자가 작성 중인 값은 불필요한 re-render나 query 갱신으로 초기화하지 않는다.

## 14. 사진

- 사진 선택 권한은 기능을 실제로 사용할 때 요청한다.
- 권한 요청 전에 사진이 선택 사항임을 알린다.
- 권한을 거부해도 기록 흐름을 계속 사용할 수 있어야 한다.
- 가능하면 업로드 전 크기와 해상도를 줄이되 과도한 품질 저하는 피한다.
- 허용 형식과 최대 크기는 서버 계약에 맞춘다.
- 로컬 URI, 업로드 중 상태, 원격 URL을 서로 다른 타입으로 다룬다.
- 업로드 중 앱이 background로 가거나 실패하는 경우를 처리한다.
- 사진 업로드 실패 시 다음 선택지를 제공한다.
  - 다시 시도
  - 사진 없이 기록 저장
  - 사진 선택 취소
- 이미 스티커 기록 저장이 성공했다면 사진 실패 때문에 성공 상태를 되돌리지 않는다.
- 임시 파일은 성공, 취소, 로그아웃 시 정리한다.
- 사진 URL이나 서명 URL을 로그와 분석 이벤트에 남기지 않는다.

## 15. 그룹과 초대

### 그룹 생성

- 그룹명, 자유 규칙, 주간 방어권 횟수만 요구한다.
- 필드 제한은 API 계약과 동일하게 적용한다.
- 생성 성공 후 공유 링크를 제공하되 OS 공유 시트를 사용한다.
- 공유를 취소해도 그룹 생성은 유지한다.
- 생성 요청 중복 제출을 방지한다.

### 그룹 참여

- 초대 코드·링크를 받으면 먼저 그룹명과 규칙을 미리 보여준다.
- 참여 전에 멤버 개인정보를 과도하게 노출하지 않는다.
- 다음 오류를 구분해 안내한다.
  - 잘못된 코드
  - 만료된 코드
  - 이미 참여한 그룹
  - 삭제되었거나 접근할 수 없는 그룹
- 오류가 있어도 직접 코드 입력이나 홈 이동 등 다음 행동을 제공한다.
- 참여 성공 후 그룹 달력으로 이동하고 그룹 목록 query를 갱신한다.

### 권한 UI

- 방장에게만 그룹 규칙과 방어권 설정 진입점을 보여준다.
- UI에서 버튼을 숨기는 것만으로 권한을 보장했다고 간주하지 않는다.
- 서버의 `403` 또는 권한 변경 응답을 처리하고 최신 그룹 정보를 다시 불러온다.
- 일반 멤버에게 관리 버튼을 잠긴 상태로 노출해 혼란을 만들지 않는다.

## 16. 반응

- 서버에서 확정한 고정 이모지 5종만 표시한다.
- 본인 기록에는 반응 바를 렌더링하지 않는다.
- 현재 사용자가 선택한 반응과 전체 개수를 구분해 표시한다.
- 한 번 탭하면 추가, 다시 탭하면 취소한다.
- 빠른 연속 탭과 느린 네트워크에서도 최종 상태가 사용자의 마지막 의도와 일치해야 한다.
- 낙관적 업데이트 실패 시 이전 상태로 복원하고 가벼운 오류 안내를 제공한다.
- 반응 실패가 기록 상세 전체를 오류 화면으로 바꾸지 않게 한다.
- 이모지 자체만으로 의미를 전달하지 말고 접근성 label을 제공한다.

## 17. 스트릭과 방어권 UI

- 서버가 계산한 값을 표시하고 클라이언트에서 별도의 진실 값을 만들지 않는다.
- 개인 뷰에서는 개인 스트릭, 그룹 뷰에서는 해당 그룹 스트릭 하나만 표시한다.
- 현재 스트릭, 최장 기록, 누적 기록일을 혼동하지 않도록 라벨을 명시한다.
- 방어권 자동 사용은 사후에 중립적인 안내로 보여준다.
- 방어권이 없거나 스트릭이 끊겨도 붉은 경고, 실패 아이콘, 비난 문구를 사용하지 않는다.
- 오늘 미기록 상태를 이미 끊긴 스트릭처럼 표현하지 않는다.
- 주간 방어권 리셋 시점과 남은 수량은 서버 응답을 기준으로 표시한다.
- 스트릭 로딩 실패가 달력 기록 기능을 막지 않게 한다.

## 18. 로딩·빈 상태·오류

모든 주요 화면에는 다음 상태가 있어야 한다.

### 로딩

- 첫 로딩에는 레이아웃이 크게 흔들리지 않는 skeleton 또는 안정적인 placeholder를 사용한다.
- 버튼 하나의 작업은 전체 화면 spinner보다 해당 버튼 진행 상태를 우선한다.
- 캐시 데이터가 있으면 재검증 중에도 가능한 범위에서 기존 화면을 유지한다.

### 빈 상태

- 개인 기록 없음: 오늘 칸과 첫 스티커 기록 행동을 강조한다.
- 그룹원이 나뿐임: 초대 링크 공유 행동을 제공한다.
- 그룹 기록 없음: 경고 없이 첫 기록을 제안한다.
- 반응 없음: “아직 반응이 없어요” 정도의 중립적 표현을 사용한다.
- 빈 상태를 서버 오류와 같은 화면으로 처리하지 않는다.

### 오류

- 네트워크 오류: 연결 확인과 재시도
- 인증 만료: 세션 복구 또는 로그인 이동
- 권한 없음: 가능한 이전 화면으로 복귀
- 초대 오류: 잘못됨과 만료를 구분
- 사진 오류: 기록 계속 진행
- 서버 오류: 기술 용어 없이 재시도와 안전한 이동 제공

Toast/Snackbar는 짧은 결과 알림에 사용한다. 사용자가 결정을 내려야 하거나 입력을 수정해야 하는 오류를 사라지는 Toast에만 넣지 않는다.

## 19. 디자인 시스템과 문구

### 디자인 토큰

- 색상, 간격, radius, typography, shadow, icon size를 theme token으로 관리한다.
- 화면에 임의의 hex 색상과 반복 숫자를 흩뿌리지 않는다.
- 라이트·다크 모드 지원 범위는 제품 결정에 따르되 색상을 직접 고정하지 않는다.
- 상태 표현을 색상 하나에만 의존하지 않는다.

### 공통 컴포넌트

최소 다음 UI를 공통화한다.

- Button
- IconButton
- TextField
- PasswordField
- BottomSheet
- Dialog
- Screen
- LoadingState
- EmptyState
- ErrorState
- StickerChip
- CalendarDayCell
- MemberRecordItem
- ReactionButton
- StreakSummary

공통화는 실제 반복과 동일한 상호작용이 확인된 뒤 수행한다. 모든 화면을 거대한 범용 컴포넌트의 옵션으로 만들지 않는다.

### 문구

- 한국어를 기본으로 하되 사용자 노출 문자열을 컴포넌트에 직접 흩뿌리지 않는다.
- 강압적 명령, 죄책감, 실패 강조를 피한다.
- “오늘도 안 했어요”, “스트릭 실패” 대신 현재 상태와 가능한 다음 행동을 말한다.
- 오류 메시지는 원인과 복구 행동을 함께 제공한다.
- 서버 메시지를 그대로 사용자에게 출력하지 않는다.
- 느낌표와 경고색을 과도하게 사용하지 않는다.

## 20. 접근성

- 모든 터치 대상은 최소 44×44pt 수준을 확보한다.
- 아이콘 전용 버튼에 접근성 label과 role을 제공한다.
- 스티커와 이모지는 시각 정보 외에 읽을 수 있는 이름을 가진다.
- 달력 날짜 셀은 날짜, 기록 여부, 스티커, 오늘 여부를 한 문장으로 읽을 수 있어야 한다.
- 선택 상태를 색상만으로 표현하지 않는다.
- 텍스트 확대 시 잘리거나 버튼이 겹치지 않게 한다.
- 고정 높이 텍스트 컨테이너를 피한다.
- 화면 전환, 오류 발생, 저장 성공 등 중요한 변화는 스크린 리더에 전달한다.
- 키보드가 열린 상태에서 포커스 이동과 제출이 가능해야 한다.
- 모션 감소 설정을 존중하고 필수 정보에 애니메이션을 의존하지 않는다.
- 대비가 낮은 연한 색만으로 빈 상태와 비활성 상태를 표현하지 않는다.

## 21. 성능

- FlatList/SectionList가 필요한 목록을 ScrollView 안에 무리하게 중첩하지 않는다.
- 멤버와 기록 목록은 안정적인 ID를 key로 사용한다.
- 큰 이미지는 표시 크기에 맞춰 요청하고 캐시 정책을 사용한다.
- 한 화면 진입에 같은 API를 중복 호출하지 않는다.
- query `staleTime`과 캐시 정책은 데이터 성격에 따라 정한다.
- 앱 전역 상태 변경으로 달력 전체가 다시 렌더링되지 않게 한다.
- `useMemo`, `useCallback`, `memo`는 측정 또는 명확한 referential stability 필요가 있을 때 사용한다.
- JS thread를 오래 막는 달력 집계나 이미지 처리를 피한다.
- 개발 중 느린 기기와 느린 네트워크 조건을 확인한다.
- 애니메이션은 기록 속도를 방해하지 않는 짧은 피드백으로 제한한다.

## 22. 오프라인과 네트워크

MVP에서는 복잡한 오프라인 쓰기 동기화를 구현하지 않는다.

- 성공적으로 조회한 달력은 query cache 범위에서 다시 보여줄 수 있다.
- 오프라인 캐시 데이터에는 마지막 갱신 상태를 구분할 수 있게 한다.
- 오프라인에서 mutation을 성공으로 가장하지 않는다.
- 기록 작성 중 네트워크가 끊기면 입력값을 유지하고 재시도할 수 있게 한다.
- 앱 재시작을 넘는 작성 임시 저장은 별도 요구가 있을 때 구현한다.
- 온라인 복귀 시 무조건 모든 query를 동시에 갱신하지 않는다.
- 중복 저장 위험이 있는 mutation queue를 임의로 만들지 않는다.

## 23. 딥링크와 공유

- 초대 링크의 scheme과 universal/app link 구조를 한곳에서 관리한다.
- 앱이 설치되지 않은 경우의 웹 fallback은 배포 전에 확정한다.
- 로그인 전 초대 링크를 잃지 않고 인증 후 복원한다.
- 초대 링크에서 직접 그룹 참여를 완료하지 않고 미리보기를 거친다.
- 외부 입력인 URL과 코드는 검증한 후 라우팅한다.
- 공유 문구에는 토큰, 내부 ID, 불필요한 개인정보를 포함하지 않는다.
- 링크 중복 수신으로 같은 화면이 여러 번 쌓이지 않게 한다.

## 24. 알림과 권한

- 알림은 MVP 막바지에 구현한다.
- 첫 실행 직후 알림 권한을 요청하지 않는다.
- 리마인더의 가치와 빈도를 설명한 뒤 사용자가 켤 때 권한을 요청한다.
- 알림은 기본적으로 끄기 쉽게 만든다.
- 기록을 재촉하거나 죄책감을 주는 문구를 사용하지 않는다.
- 알림 탭 시 관련 달력 또는 기록 화면으로 안전하게 이동한다.
- 이미 삭제되거나 접근할 수 없는 대상이면 홈으로 fallback한다.
- 디바이스 토큰 등록·갱신·해제를 세션 생명주기와 연결한다.
- 로그아웃 후 이전 사용자의 알림 대상이 남지 않게 한다.
- 카메라, 사진, 알림 권한 거부를 앱 사용 차단으로 처리하지 않는다.

## 25. 보안과 개인정보

- 앱 번들에 비밀키를 넣지 않는다. 모바일 환경 변수는 비밀 저장소가 아니다.
- 공개 가능한 API base URL과 공개 식별자만 빌드 설정에 둔다.
- 인증 토큰, 비밀번호, 소감, 사진 URL, 초대 코드를 console에 출력하지 않는다.
- production build에서 불필요한 디버그 로그를 제거한다.
- 사용자 입력을 오류 리포트와 분석 도구로 자동 전송하지 않는다.
- 외부 URL은 허용 scheme과 host를 검증한 뒤 연다.
- 서버가 소유권과 권한을 검사하더라도 클라이언트에서도 잘못된 행동 진입점을 만들지 않는다.
- 개인정보를 화면 캡처로부터 보호해야 하는 범위는 별도 제품·보안 결정 후 적용한다.
- 클립보드 복사는 꼭 필요한 초대 코드·링크에만 제공하고 완료를 알린다.

## 26. 분석 이벤트

MVP의 검증 질문에 필요한 최소 이벤트만 수집한다.

권장 이벤트 예시:

- 가입 완료
- 첫 개인 기록 완료
- 기록 생성·수정
- 그룹 생성
- 초대 공유
- 초대 미리보기
- 그룹 참여
- 반응 추가·취소
- 달력 전환
- 리마인더 opt-in·opt-out

규칙:

- 이벤트 이름과 속성은 한곳에서 타입으로 관리한다.
- 이메일, 이름, 소감 내용, 사진 URL, 토큰, 초대 코드 원문을 전송하지 않는다.
- 화면 렌더링만으로 같은 이벤트를 반복 전송하지 않는다.
- 분석 실패가 사용자 기능을 막아서는 안 된다.
- 수집 목적이 없는 이벤트를 관성적으로 추가하지 않는다.
- 분석 공급자 도입 전 개인정보 처리방침과 동의 요구를 확인한다.

## 27. 코드 작성 규칙

- TypeScript strict mode 오류를 무시하지 않는다.
- `any`를 사용하지 않는다. 외부 입력은 `unknown`으로 받고 검증한다.
- 컴포넌트 props와 함수 반환 타입을 명확히 한다.
- enum 대신 서버 문자열 union 또는 `as const` 객체가 더 안전한 경우 이를 사용한다.
- API 응답 타입과 화면 표시 타입을 구분한다.
- 컴포넌트는 데이터 로딩, 복잡한 변환, 큰 UI를 동시에 책임지지 않는다.
- hook은 React hook 규칙을 따르고 숨겨진 전역 부작용을 만들지 않는다.
- `useEffect`를 데이터 파생과 상태 동기화의 기본 수단으로 사용하지 않는다.
- 파생 가능한 값은 state로 중복 저장하지 않는다.
- 빈 의존성 배열로 경고를 억지로 피하지 않는다.
- style object를 큰 컴포넌트 안에 반복 생성하지 않는다.
- 플랫폼 차이는 `.ios.tsx`, `.android.tsx` 또는 명시적 adapter로 격리한다.
- 한국어 도메인 의미가 드러나는 명확한 영어 이름을 사용한다.
- `util`, `helper`, `manager`, `data`처럼 범위가 모호한 이름을 피한다.
- 주석은 코드의 반복 설명이 아니라 제품 제약과 선택 이유를 기록한다.
- TODO에는 이유와 완료 조건을 적고 방치하지 않는다.

## 28. 테스트

기능 구현은 테스트를 포함해야 완료다.

### 단위 테스트

- 날짜 문자열 변환
- 월 달력 셀 생성
- 월·연도·윤년 경계
- API DTO → ViewModel 변환
- 오류 코드 → 사용자 상태 변환
- query key factory
- 기록 폼 validation

시간에 의존하는 테스트는 시스템 현재 시각을 직접 사용하지 않고 고정된 clock/date adapter를 사용한다.

### 컴포넌트 테스트

React Native Testing Library로 사용자 행동 중심으로 검증한다.

- 스티커를 선택해야 저장 버튼이 활성화된다.
- 소감과 사진 없이 저장할 수 있다.
- 기존 기록은 수정 상태로 열린다.
- 본인 기록에는 수정·삭제만 보인다.
- 다른 사람 기록에는 반응 바만 보인다.
- 그룹원이 나뿐이면 초대 행동이 보인다.
- 빈 달력은 오늘 기록 행동을 제공한다.
- 사진 권한 거부 후에도 기록할 수 있다.
- 오류 후 입력값이 유지된다.
- 접근성 label로 주요 행동을 찾을 수 있다.

구현 세부사항, hook 호출 횟수, 내부 state를 직접 검증하지 않는다.

### 통합 테스트

API 계층을 제어 가능한 mock으로 대체하여 다음 흐름을 검증한다.

- 로그인과 세션 복구
- 첫 개인 기록
- 기록 생성 후 달력·스트릭 갱신
- 그룹 생성과 공유
- 초대 링크 수신 → 로그인 → 미리보기 → 참여
- 반응 추가·취소와 실패 롤백
- 토큰 만료와 단일 refresh
- 로그아웃 후 사용자 cache 제거

### E2E

실제 iOS·Android 환경에서 최소 다음 happy path를 유지한다.

1. 가입 또는 테스트 사용자 로그인
2. 개인 달력 첫 기록
3. 기록 수정
4. 그룹 생성
5. 다른 사용자 참여
6. 그룹 기록 확인
7. 반응 추가·취소

핵심 흐름은 접근성 label이나 안정적인 testID를 사용한다. 화면 텍스트 전체나 좌표에 과도하게 의존하지 않는다.

### 버그 수정

- 먼저 실패를 재현하는 테스트를 추가한다.
- 수정 후 관련 회귀 테스트와 전체 타입 검사를 실행한다.
- iOS와 Android 중 한 플랫폼에만 발생하는 버그는 해당 차이를 문서화한다.

## 29. 환경 설정과 빌드

- 개발, 스테이징, 운영 API 환경을 분리한다.
- 환경별 앱 식별자, 딥링크, 소셜 로그인 redirect URI를 명시적으로 관리한다.

- 앱 시작 실패 시 원인을 찾을 수 있는 안전한 오류 리포팅을 제공한다.

- iOS와 Android에서 핵심 상호작용이 동작한다.
- 스티커만으로 기록할 수 있다.
- 선택 기능 실패가 핵심 기록 흐름을 막지 않는다.
- 서버 상태, 화면 상태, 인증 상태가 중복 저장되지 않는다.
- TypeScript strict 검사에 통과한다.
- lint와 테스트가 통과한다.
- 주요 변경에 컴포넌트 또는 흐름 테스트가 있다.
- 키보드, 접근성 label, 터치 영역을 확인했다.
- 사용자 데이터나 비밀값을 로그에 남기지 않는다.
- API·라우트·환경 설정 변경이 문서화되어 있다.
- MVP 밖 기능이나 불필요한 의존성을 추가하지 않았다.
- 실제 기기에서 확인하지 못한 항목은 결과에 명시한다.

단계별 완료 기준:

- 기반: 로그인 후 기본 개인 달력으로 진입한다.
- 개인 기록: 그룹 없이 스티커 하나로 생성·조회·수정할 수 있다.
- 스트릭: 현재 달력에 맞는 스트릭과 방어권을 압박 없이 보여준다.
- 그룹: 두 명 이상이 그룹을 만들고 참여해 서로의 기록을 볼 수 있다.
- 반응: 다른 멤버 기록에 반응을 추가·취소하고 본인 기록에는 반응할 수 없다.
- 통합 뷰: 여러 그룹 기록의 출처를 구분하며 달력을 전환할 수 있다.
- 사진·마무리: 사진 실패, 권한 거부, 빈 데이터, 잘못된 초대, 네트워크 오류에서도 기록 흐름이 유지된다.

## 32. 금지 사항

- 가입 직후 그룹 참여를 강제하지 않는다.
- 스티커 외 입력을 기록 저장 필수 조건으로 만들지 않는다.
- 미기록일, 스트릭 중단, 방어권 소진을 실패나 경고로 과장하지 않는다.
- 본인 기록에 반응 UI를 보여주지 않는다.
- 다른 사람의 기록에 수정·삭제 UI를 보여주지 않는다.
- 서버 응답 데이터를 여러 전역 store에 복제하지 않는다.
- 화면 컴포넌트에서 직접 API와 보안 저장소를 호출하지 않는다.
- 날짜 전용 값을 무조건 JavaScript `Date`로 변환하지 않는다.
- 토큰이나 민감 정보를 AsyncStorage, 로그, 분석 이벤트에 저장하지 않는다.
- 사진 업로드 실패로 이미 성공한 기록을 실패 처리하지 않는다.
- API 오류 메시지를 가공 없이 사용자에게 표시하지 않는다.
- 권한 요청을 첫 실행에 몰아서 하지 않는다.
- 접근성 label 없이 이모지·아이콘만으로 행동을 제공하지 않는다.
- 테스트를 통과시키기 위해 제품 권한이나 입력 규칙을 약화하지 않는다.
- MVP에 필요하지 않은 채팅, 랭킹, 배지, 통계, AI 기능을 선제 구현하지 않는다.
- 모바일 앱에 서버 비밀키나 관리자 권한 키를 포함하지 않는다.
