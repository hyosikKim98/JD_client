# 「뭐 했음 청년」 제품 기획서

> 퇴근 후 자기개발하는 직장인을 위한, 강제 없는 기록·응원 중심 습관 기록 앱

- 문서 상태: MVP 기획 1차 완료
- 확정 범위: 문제 정의부터 위험 요소까지 1차 검토 완료

## 1. 해결하려는 문제

자기개발 의지가 있는 직장인은 퇴근 후 혼자 무언가를 시도하지만, 혼자서는 꾸준함이 무너지기 쉽다. 기존 습관 앱(열품타·챌린저스 등)은 시간 측정·벌금·인증 압박처럼 강제력이 강해 부담스럽고, 그 결과 **기록조차 부담스러워 하지 않게 되는 악순환**에 빠진다.

### 해결 방향

마찰이 거의 없는 기록(스티커 하나)과 강제 없는 그룹의 가벼운 사회적 동기부여로 꾸준함을 떠받친다.

## 2. 목표 사용자

퇴근 후 운동·어학·독서·사이드 프로젝트·자격증 등의 자기개발에 임하는 직장인이다. 절박함보다 향상심으로 움직이며, 벌칙·경쟁보다 느슨한 응원과 함께하는 감각을 원하는 사람을 대상으로 한다.

핵심 사용 단위는 친구·동료로 구성된 소규모 그룹이다. 다만 혼자서도 개인 달력만으로 최소 가치를 얻을 수 있어야 한다. 이는 그룹이 없는 초기 사용자를 위한 콜드스타트 안전망이다.

## 3. 핵심 가치

1. **마찰 없는 기록**
   - 스티커 하나로 오늘의 자기개발을 기록한다.
   - 글과 사진은 선택 사항이다.

2. **강제 없는 동기부여**
   - 벌금·랭킹·의무 인증을 사용하지 않는다.
   - 그룹의 기록 공유, 반응, 방어권이 있는 스트릭으로 부드럽게 지속을 돕는다.

3. **쌓이는 성취의 가시화**
   - 흩어진 하루하루의 기록을 달력과 연속 기록으로 보여준다.

## 4. 사용자 이용 흐름

```text
가입/로그인
  ↓
첫 진입 — 개인 달력이 기본 화면
  ↓
추천 스티커 세트 노출
  ↓
오늘 날짜에 스티커 하나를 붙여 첫 기록 완료
  ↓
[선택] 그룹 생성 또는 초대 코드로 참여
  ↓
그룹 달력에서 내 기록과 그룹원 기록 확인
  ↓
그룹원 기록에 이모지 반응 / 내 기록에 반응 받기
  ↓
다음 날부터 스트릭·방어권 누적 → 지속 동기 형성
```

### 설계 의도

그룹 가입을 기록보다 뒤에 둔다. **혼자 한 번 기록한 뒤 마음에 들면 그룹으로 이동하는 순서**를 통해 빈 그룹에 진입한 사용자의 이탈을 막는다.

### 핵심 예외 처리

- 빈 데이터
  - 혼자뿐인 그룹에서는 초대 유도 화면을 보여준다.
  - 기록이 0개라면 오늘 칸을 강조하고 스티커 하나만 붙이도록 유도한다.
- 기록 누락
  - 붉은 경고 등 죄책감을 유발하는 표현을 사용하지 않는다.
  - 방어권이 있으면 자동 사용하고 사후 알림을 제공한다.
  - 방어권이 없으면 현재 스트릭은 끊기지만 누적·최장 기록은 보존한다.
- 권한
  - 방장만 그룹 규칙과 방어권을 설정한다.
  - 사용자는 본인 기록만 수정·삭제할 수 있다.
  - 다른 사용자의 기록에는 반응만 할 수 있다.
- 오류
  - 사진 업로드가 실패해도 기록 자체는 저장한다.
  - 초대 코드가 잘못되었거나 만료되었다면 구체적인 사유를 표시한다.

## 7. 화면 구성

### 로그인·가입

- 소셜 로그인과 자체 가입을 지원한다.
- 자체 가입에서는 이메일 형식 아이디를 사용한다.
- 입력 항목은 최소화한다.

### 개인 달력(홈)

- 월 달력
- 일자별 내 스티커
- 오늘 칸 강조
- 현재 스트릭·방어권 표시
- 그룹 전환

### 기록 작성(바텀 시트)

- 스티커 선택: 추천 5종, 커스텀 스티커, 스티커 추가
- 한 줄 소감
- 사진
- 스티커만 선택해도 저장 버튼을 활성화한다.
- 기존 기록이 있으면 수정 모드로 연다.

### 그룹 공유 달력

- `(스티커)(이름)`을 세로로 나열하고 펼쳐 본다.
- 그룹명과 자유 규칙란을 표시한다.
- 멤버 목록과 초대 기능을 제공한다.

### 기록 상세

- 스티커, 이름, 날짜, 소감, 사진을 표시한다.
- 다른 사용자의 기록에는 고정 이모지 5종 반응 바를 표시한다.
- 본인 기록에는 반응 바 대신 수정·삭제 기능을 표시한다.

### 그룹 생성

- 그룹명
- 자유 규칙란
- 주간 방어권 횟수
- 생성 즉시 공유 링크 제공

### 그룹 참여

- 초대 코드 또는 링크로 접근한다.
- 그룹 규칙을 미리 확인한 후 참여한다.

### 멤버 목록

- 멤버별 최장 기록을 표시한다.
- 방장은 멤버·기록 관리 기능에 접근할 수 있다.

### 스티커 관리

- 커스텀 스티커를 추가·편집한다.

### 달력 표기 규칙

- 그룹 공유 달력: `(스티커)(멤버 이름)` — 누가 기록했는지 보여준다.
- 개인 통합 달력: `(스티커)(출처: 그룹명 또는 "나의 기록")` — 어디에 기록했는지 보여준다.
- 여러 그룹에 가입할 수 있다.
- 기본 달력을 지정하면 첫 화면에 해당 달력을 표시한다.
- 상단에 그룹 전환 UI를 제공한다.

## 8. 데이터 구조

### 8.1 핵심 엔티티와 관계

#### User

- `id`
- `name`
- `email`: 자체 가입 아이디
- 로그인 정보
- `created_at`
- `default_calendar_id`

개인정보는 이름만 추가 수집한다.

#### Group

- `id`
- `name`
- `rule_text`
- `weekly_defense_count`
- `owner_id`
- `invite_code`
- `created_at`

개인 달력인 **나의 기록**도 `Group`의 한 종류로 가입 시 자동 생성한다.

#### Membership

`User`와 `Group` 사이의 N:M 관계다.

- `user_id`
- `group_id`
- `role`: 방장 또는 멤버
- `joined_at`
- 탈퇴 시 기록 보존·삭제 선택값

#### Sticker

- `id`
- `name`
- `icon` 또는 `emoji`
- `group_id`

스티커는 그룹에 종속된다. 기본 5종은 그룹 생성 시 복제하고, 커스텀 스티커는 해당 그룹에 추가한다. 개인 커스텀 스티커는 **나의 기록** 그룹에 종속된다.

#### Record

- `id`
- `user_id`
- `group_id`
- `sticker_id`
- `record_date`
- `memo`: nullable
- `photo_url`: nullable
- `created_at`
- `updated_at`

제약 조건:

- `(user_id, group_id, record_date)`는 유일해야 한다.
- 이를 통해 그룹별 하루 1스티커를 강제한다.
- 같은 날짜에 다시 기록하면 새 행을 생성하지 않고 기존 행을 수정한다.

#### Reaction

- `id`
- `record_id`
- `from_user_id`
- `emoji`: 고정 5종 중 하나

제약 조건:

- `(record_id, from_user_id, emoji)`는 유일해야 한다.
- 동일 반응은 토글 방식으로 추가·삭제한다.
- 본인 기록에는 반응할 수 없다.

#### Streak / Defense

- 개인 스트릭: `User` 단위
- 그룹 스트릭: `Membership` 단위
- 스트릭은 기록 날짜에서 파생 계산하며 필요하면 캐시한다.
- 방어권은 각 스트릭에서 독립적으로 소모한다.

### 8.2 핵심 모델 원칙

- 모든 기록은 **나의 기록**을 포함해 항상 하나의 `Group`에 속한다.
- `Record`는 항상 `group_id`를 가진다.
- 하루 1스티커는 그룹 달력별 규칙이다.
- 개인 통합 달력은 모든 그룹 기록의 합집합이며 직접 기록도 가능하다.
- 스트릭은 현재 보고 있는 달력을 기준으로 하나만 표시한다.
  - 개인 뷰: 개인 스트릭
  - 그룹 뷰: 그룹 스트릭
- 개인 스트릭과 그룹 스트릭 데이터는 모두 저장한다.

## 9. 기술 구성

| 영역 | 선택 | 이유 | 단점 |
| --- | --- | --- | --- |
| 클라이언트 | React Native | iOS·Android 단일 코드베이스, 넓은 인력 풀, 무거운 그래픽이 필요하지 않음 | 일부 네이티브 성능 제약이 있으나 이 제품에는 영향이 작음 |
| 백엔드 | Spring Boot 모놀리식 REST API | RDB·트랜잭션·Security 생태계가 탄탄하고 디버깅이 단순함 | 초대형 트래픽에서는 확장 한계가 있으나 현재 범위에는 해당하지 않음 |
| DB | PostgreSQL | 관계와 유일 제약이 핵심인 모델에 적합함 | NoSQL보다 스키마 변경이 번거로움 |
| 이미지 | 오브젝트 스토리지(S3 등) + CDN | DB에는 URL만 저장하며 사진은 부가 기능이라 비중이 작음 | 외부 비용과 키 관리가 필요함 |
| 인증 | 소셜 로그인 + 자체 가입(이메일 아이디, JWT) | 소셜 로그인으로 진입 마찰을 줄이고 소셜 계정이 없는 사용자도 지원함 | 자체 가입은 비밀번호 관리 책임이 생김 |
| 알림 | MVP 막바지 무강제형 리마인더 | 재방문 계기를 제공함 | 과하면 피로하므로 쉽게 끌 수 있어야 함 |

### 의도적으로 배제한 기술

- MSA·Kafka: 현재 규모와 비동기 처리 수요에 불필요하다.
- AI 요약: 출시 후 확장 기능이다.
- 실시간 WebSocket: 폴링으로 충분하다.

| 2. 스트릭·방어권 | 개인 스트릭 계산, 방어권 자동 사용·사후 알림, 상단 표시 | 연속일 증가와 끊김이 정의된 규칙대로 동작한다. |
| 3. 그룹 | 그룹 생성·초대·참여, 공유 달력, 멤버 목록, 최장 기록 | 두 명 이상이 한 그룹에서 서로의 기록을 확인할 수 있다. |
| 4. 반응 | 고정 이모지 5종 토글, 받은 반응 표시, 본인 기록 반응 차단 | 그룹원끼리 반응을 주고받을 수 있다. |
| 5. 통합 뷰·다중 그룹 | 통합 달력, 다중 그룹, 기본 달력, 달력 전환 | 여러 그룹이 있을 때 통합과 전환이 자연스럽게 동작한다. |
| 6. 사진·다듬기·리마인더 | 사진 업로드, 온보딩 톤, 예외 점검, 가벼운 리마인더 | 오류·빈 데이터·권한 상황에서 서비스가 무너지지 않는다. |

### 개발 원칙

그룹 기능이 핵심 문제를 검증하는 단계이므로 개인 기능인 1~2단계는 최소 범위로 빠르게 끝내고 3단계에 자원을 집중한다. 각 단계는 독립적으로 사용할 수 있는 상태로 마감한다.

## 11. 위험 요소

### 11.1 제품·시장 위험

#### 콜드스타트

- 문제: 직장인은 함께할 그룹을 구하기 어려울 수 있다.
- 대응:
  - 개인 기록만으로도 가치를 보장한다.
  - 그룹 생성 즉시 공유 링크를 제공한다.
  - 친한 사람 2~3명으로 구성된 소규모 그룹부터 시작한다.

#### 차별화가 톤에 의존

- 문제: 기능은 일반 체크 앱과 겹치며, 무강제와 **잘 쉬었음** 같은 차별점은 분위기에 가까워 모방하기 쉽고 첫눈에 드러나지 않는다.
- 대응: 온보딩과 카피 전반에서 **다그치지 않는 곳**이라는 메시지를 일관되게 전달한다.

#### 무강제와 리텐션의 긴장

- 문제: 강제성이 없으면 기록을 중단하기도 쉽다.
- 대응: MVP 검증 지표를 **강제 없이 몇 주 동안 리텐션이 유지되는가**로 명확히 설정한다.

### 11.2 설계 복잡도

#### 스트릭 2종과 방어권 2층

- 개인·그룹 스트릭을 모두 저장하지만 화면에는 현재 달력 기준 하나만 표시한다.
- 로직이 지나치게 복잡하거나 베타 사용자에게 혼란을 주면 개인 스트릭 하나로 단순화할 수 있어야 한다.

#### 그룹별 스티커 중복 어휘

- 통합 뷰나 통계 확장 시 동일한 의미의 스티커가 그룹별로 중복될 수 있다.
- MVP에서는 허용하고 확장 단계에서 정규화를 검토한다.

### 11.3 운영·기술 위험

#### 사진·UGC 모더레이션

- 신고와 필터링은 MVP에서 제외하고 출시 후 확장한다.

#### 개인정보·약관

- 이름만 추가 수집하여 개인정보 부담을 줄인다.
- 출시 전 표준 이용약관과 개인정보 처리방침이 필요하다.

#### 푸시 부재로 인한 재방문 약화

- MVP 막바지에 무강제형 가벼운 리마인더를 추가한다.
